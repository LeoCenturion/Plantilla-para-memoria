\section{Conjunto de Datos y Preprocesamiento}

Esta sección describe el conjunto de datos utilizado y las estrategias de preprocesamiento aplicadas para el desarrollo del trabajo. Se detallan las consideraciones sobre la frecuencia de los datos, las técnicas de construcción de barras y el análisis de la importancia de las características.

\subsection{Conjunto de datos}

El trabajo se basó en el análisis de datos del mercado spot de Bitcoin. El mercado spot se define como aquel en el que los activos financieros, en este caso Bitcoin (BTC) frente al Tether (USDT), se compran y venden para su liquidación y entrega inmediata. Las variables disponibles para este mercado incluyen el precio de apertura, el precio de cierre, el valor mínimo y máximo alcanzado, el volumen comercializado en unidades de Bitcoin y en USDT, y la hora exacta en la que se tomó cada medición.

Estos datos se recopilaron desde el 1 de enero de 2020 a través de la plataforma de Binance. Inicialmente, se experimentó con una frecuencia de una hora, la cual presentó la desventaja de acotar el volumen de datos disponibles para el entrenamiento del modelo. Además, se observó que las estrategias probadas con esta frecuencia arrojaban peores resultados. Se infirió que esto se debía, posiblemente, a la menor cantidad de datos disponibles, lo que limitó la capacidad del modelo para aprender patrones robustos. Por lo anterior, se optó por utilizar una frecuencia de un minuto, lo que generó una mejora significativa en el desempeño de las estrategias estudiadas al proporcionar un conjunto de datos más denso y amplio.

\subsection{Estrategias de preprocesamiento}

Para el preprocesamiento de los datos, se aplicaron diversas estrategias con el fin de optimizar su calidad y relevancia para el modelo:

\begin{itemize}
    \item Conversión a micro-Bitcoins (uBTC): se realizó una conversión de BTC a uBTC, una práctica estándar en el análisis de criptomonedas para manejar unidades más pequeñas y facilitar los cálculos.
    \item Agrupación por Volumen (Volume Bars): se exploraron formas de agrupar los datos no por intervalos de tiempo fijos, sino por volumen en unidades constantes tanto de BTC como de USDT. Adicionalmente, se probaron las volume imbalance bars y las tick imbalance bars. Las volume imbalance bars se construyen cuando se alcanza un volumen neto desequilibrado entre compras y ventas, y las tick imbalance bars se forman con un desequilibrio neto en el número de ticks (cambios de precio). Después de una investigación exhaustiva de la literatura y pruebas empíricas, se decidió continuar con la agrupación de barras por volumen fijo. Esta elección se fundamentó en que las barras de volumen aportan una mejora notable en las propiedades estadísticas de la distribución del precio de Bitcoin, ya que reducen la autocorrelación y la heterocedasticidad, lo que las hace más adecuadas para el entrenamiento de modelos de machine learning. Se adoptó un límite de volumen de 50.000 uBTC, el cual proporcionó un equilibrio satisfactorio entre la cantidad de puntos de datos obtenidos y la reducción de la varianza correspondiente a cada barra.
\end{itemize}

\subsection{Ingeniería de características y análisis de importancia}

Para la ingeniería de características, se generó una extensa lista de indicadores técnicos. Adicionalmente, se hizo uso de técnicas de reducción de dimensionalidad como el Análisis de Componentes Principales (PCA). Para determinar las características más relevantes para el modelo final, se realizó un análisis de importancia utilizando un conjunto de métricas, entre ellas: la Disminución Media de Impureza (MDI), la Disminución Media de Exactitud (MDA) y la Importancia de Característica Individual (SFI). También se analizaron las funciones de autocorrelación (ACF) y autocorrelación parcial (PACF) de los componentes principales para identificar dependencias temporales.

\section{Estrategias de referencia (Baselines)}

El trabajo de Guilherme Palazzo, titulado ``Predicting BTC cryptocurrency price movement in a pre-defined trading volume window using Machine Learning models'', se centra en predecir la dirección del movimiento del precio de Bitcoin (BTC) al formular el problema como una tarea de clasificación en lugar de intentar pronosticar el precio exacto.

Principales componentes del trabajo de Palazzo:

\begin{itemize}
        \item Muestreo basado en volumen (Volume bars): se argumenta que los mercados no procesan la información en intervalos de tiempo constantes (lo que causa submuestreo en alta actividad y sobremuestreo en baja actividad), Palazzo rechaza las barras de tiempo tradicionales. En su lugar, agrupa los datos en barras de volumen (específicamente ventanas de 50,000 unidades de BTC).
        \item Etiquetado dinámico: se inspira directamente en las metodologías de Marcos López de Prado (como el enfoque de la triple barrera) y no asume un marco de tiempo fijo ni distribuciones predefinidas. Se etiquetan los datos al evaluar si el retorno futuro supera un umbral dinámico dictado por la desviación estándar local (volatilidad, $\sigma$) multiplicada por una constante predefinida ($	au$). Para la corrección del desbalance natural de las clases resultantes, se utiliza el algoritmo K-Nearest Neighbors (KNN) para hacer undersampling en los datos de entrenamiento.
        \item Ingeniería y selección de características: En lugar de operar con precios crudos, se utilizan retornos logarítmicos, se calculan z-scores y se generan múltiples indicadores de análisis técnico (tendencia, volumen y momentum). Después de escalar estas características, se emplea el Criterio de Información Mutua (MIC) para seleccionar las 40 variables con mayor poder predictivo. En el caso de la Regresión Logística, se aplica previamente el Factor de Inflación de la Varianza (VIF) para eliminar la multicolinealidad.
        \item Modelado riguroso y resultados: entrenó y evaluó tres modelos predictivos: Regresión Logística, Long-Short Term Memory (LSTM) y eXtreme Gradient Boosting (XGBoost). El XGBoost fue el mejor algoritmo, el cual logró un Área Bajo la Curva (AUC) de 0.66 y un Ratio de Sharpe (anualizado) de 6.27 en el conjunto de prueba, y un Sharpe de 1.08 en la validación estrictamente fuera de muestra, superó con creces a la estrategia benchmark. Para la optimización de los hiperparámetros sin contaminar los datos (look-ahead bias), se usó TimeSeriesSplit de scikit-learn y se mantuvo la integridad cronológica del conjunto.
    \end{itemize}

Se seleccionó el trabajo de Guilherme Palazzo como punto de referencia (baseline) para esta investigación por varias razones fundamentales que garantizan una comparación rigurosa. En primer lugar, su metodología representa una aplicación empírica ideal de las teorías de López de Prado, utilizando conceptos como los relojes de volumen para mitigar la naturaleza no estacionaria del mercado y el etiquetado dinámico. Al reformular el problema como una tarea de clasificación direccional en lugar de una regresión de precios, su enfoque aísla eficazmente el ruido del mercado, proporcionando una base de datos óptimamente estructurada contra la cual medir el valor añadido de nuevas arquitecturas.

Además, su marco de prueba es estadísticamente robusto, implementando defensas estrictas contra el sobreajuste mediante el uso de validación cruzada temporalmente consciente. Finalmente, el trabajo de Palazzo ofrece un conjunto claro de métricas de rendimiento (como un AUC de 0.66 y un Ratio de Sharpe de 6.27) para modelos maduros como XGBoost y LSTM. Esto establece un límite superior de rendimiento bien definido para los algoritmos de la generación anterior, proporcionando una barrera medible que los modelos más avanzados propuestos en esta memoria debían superar.

\section{Arquitectura Transformer Propuesta}

La aplicación de la arquitectura Transformer, originalmente concebida para el Procesamiento del Lenguaje Natural (NLP), a las series temporales no fue directa. Los primeros modelos, como Informer y Autoformer, se centraron en superar la complejidad computacional cuadrática ($O(L^2)$) del mecanismo de autoatención para poder procesar secuencias largas. Informer introdujo una atención dispersa (	exttt{ProbSparse Attention}), mientras que Autoformer reemplazó la autoatención con un mecanismo de autocorrelación inspirado en el análisis clásico de series temporales.

Sin embargo, un influyente estudio \cite{Zeng:2023} demostró que modelos lineales simples podían superar a estas complejas arquitecturas en benchmarks estándar, generando un debate sobre si los Transformers eran fundamentalmente adecuados para datos temporales. La respuesta llegó con modelos como PatchTST, que revolucionaron el campo al cambiar el enfoque desde la modificación del mecanismo de atención hacia la representación de la entrada. Al segmentar la serie temporal en "parches" (subseries) que actúan como tokens, PatchTST demostró que un Transformer estándar podía superar a sus predecesores más complejos. Este avance consolidó la idea de que, con la representación de datos adecuada, los Transformers son herramientas extremadamente poderosas y sentó las bases para el desarrollo de modelos fundacionales pre-entrenados.

La elección de la arquitectura se centró en Chronos, un marco de trabajo que representa el estado del arte en modelos fundacionales pre-entrenados para series temporales. Este enfoque minimalista responde directamente al debate sobre la efectividad de los Transformers, demostrando que su poder puede ser desbloqueado al tratar las series temporales como un lenguaje. En lugar de diseñar componentes complejos y específicos para el dominio temporal, Chronos adapta las arquitecturas de modelos de lenguaje existentes con modificaciones mínimas. Dicha arquitectura se puede ver en \ref{fig:chronos_arch}.

El núcleo de Chronos es su proceso de tokenización, que convierte una serie temporal de valores reales en una secuencia de tokens discretos, haciéndola legible para un modelo de lenguaje. Este proceso consta de dos pasos:
\begin{itemize}
    \item 	Escalado (Scaling): Se aplica un "escalado por media" (mean scaling), donde cada valor de la serie se divide por la media de los valores absolutos de la serie histórica. Este método simple normaliza la serie a un rango adecuado para la cuantización y preserva los ceros, que pueden tener un significado semántico (por ejemplo, cero ventas).
    \item 	Cuantización (Quantization): La serie escalada, que sigue siendo de valor real, se convierte en tokens discretos mediante una "cuantización uniforme" (uniform binning). Los valores se asignan a uno de 	extit{B} contenedores (bins) predefinidos, cada uno representado por un token único de un vocabulario fijo. Este proceso transforma la serie de tiempo en una secuencia de "palabras", creando el "lenguaje de las series temporales".
\end{itemize}

Una vez tokenizada la serie, se utiliza una arquitectura Transformer estándar, como la familia T5 (un modelo codificador-decodificador), sin necesidad de añadir capas de descomposición de tendencia o estacionalidad. El único ajuste requerido es adaptar el tamaño del vocabulario del modelo al número de tokens de la cuantización. El modelo se entrena desde cero utilizando una función de pérdida de entropía cruzada categórica, el estándar para modelos de lenguaje, con el objetivo de predecir el siguiente token en la secuencia. Este enfoque se conoce como "regresión por clasificación", ya que el modelo aprende a predecir un valor numérico tratándolo como un problema de clasificación sobre los tokens discretos. Es importante destacar que el modelo ignora deliberadamente información externa como marcas de tiempo (timestamps), tratando la serie simplemente como una secuencia ordenada.

Para mejorar la generalización y el rendimiento en tareas de pronóstico sobre datos no vistos (zero-shot), los modelos Chronos se pre-entrenan en un corpus masivo y diverso de series temporales. Este entrenamiento se enriquece con técnicas de aumento de datos como 	TSMixup, que crea nuevas series combinando linealmente series existentes, y 	KernelSynth, que genera datos sintéticos realistas utilizando Procesos Gaussianos.

En este trabajo, en lugar de utilizar el modelo para pronóstico directo, se aprovecha su capacidad para generar representaciones ricas. Se extraen los embeddings de las capas ocultas del codificador, que son vectores de alta dimensionalidad que encapsulan la información contextual y los patrones temporales aprendidos. Estos embeddings se utilizan luego como características de entrada para un modelo tabular más tradicional y eficiente, como XGBoost, para realizar la tarea de clasificación final.

\section{Estrategia de Etiquetado para Modelos ML}

Para definir el objetivo de predicción del modelo, se evaluaron varias técnicas de etiquetado. Tras analizar los métodos de horizonte fijo y etiquetado dinámico, se adoptó el método de la barrera triple (\textit{Triple-Barrier Method}). Esta técnica, recomendada en la literatura para mercados financieros, ofrece un enfoque más adaptativo y robusto al definir las etiquetas en función de la volatilidad del activo, utilizando barreras para ganancias, pérdidas y tiempo de expiración.
Para etiquetar un punto se puede elegir una etiqueta independiente para la barrera vertical o agruparla con las barreras horizontales segun si el punto final es mayor o menor que el inicial, si es menor se agrupa con la etiqueta correspondiente a la barrera inferior y, si es mayor, con la superior. Esta alternativa permite mantener las clases a predecir balanceadas con lo cual fue la elegida.

\section{Framework de Backtesting y Evaluación}

Para asegurar una evaluación rigurosa y libre de sesgos, se implementó un motor de backtesting a medida en Python. Esta decisión se tomó debido a la necesidad de utilizar metodologías avanzadas como la validación cruzada combinatoria con purga y embargo (\texttt{Combinatorially Purged Cross-Validation}), una técnica no disponible en las bibliotecas estándar de machine learning. Este método es esencial para prevenir la fuga de información y el sobreajuste al evaluar modelos en series de tiempo financieras, garantizando que el rendimiento medido sea una estimación realista del comportamiento del modelo en datos futuros.

La evaluación de los modelos se centró exclusivamente en su rendimiento en el backtest, separando estrictamente esta fase de la optimización de hiperparámetros. Dicha búsqueda se realizó una única vez durante la etapa de investigación y validación del modelo, utilizando un conjunto de datos distinto al de la evaluación final.

Al tratarse de un problema de clasificación, y considerando el desbalance de clases inherente a las señales de trading (donde la clase "mantener" es a menudo mayoritaria), se utilizó la métrica F1 ponderada por clase para evaluar el desempeño. Esta métrica proporciona una medida balanceada entre la precisión y la exhaustividad del modelo, ajustada a la frecuencia de cada clase, lo que la hace más representativa que la exactitud simple.

\section{Modelos explorados}

Durante la fase de investigación, se exploró un amplio espectro de modelos para establecer un punto de referencia y comprender la complejidad del problema.

Inicialmente, se evaluaron estrategias de trading clásicas basadas en indicadores técnicos, como el cruce de medias móviles, las bandas de Bollinger, el MACD y las divergencias en el RSI. Si bien estos métodos son populares, demostraron ser demasiado rígidos y poco adaptativos para la alta volatilidad del mercado de criptomonedas.

Posteriormente, se probaron modelos estadísticos tradicionales para series temporales, incluyendo ARIMA, SARIMA, y variantes más complejas como ARIMAX con GARCH como variable exógena y Prophet de Facebook. Estos modelos, aunque más sofisticados, lucharon por capturar las dinámicas no lineales y no estacionarias del precio de Bitcoin.

Siguiendo la recomendación de López de Prado sobre la idoneidad de los modelos basados en árboles para datos financieros, se exploraron alternativas a XGBoost. Se utilizó la biblioteca \texttt{AutoGluon}, que permite entrenar y comparar de forma automática un conjunto de modelos tabulares de última generación, incluyendo diferentes configuraciones de redes neuronales y ensambles, para encontrar un posible reemplazo al modelo de Palazzo.

Finalmente, en cuanto a la arquitectura principal de Chronos, se experimentó con diferentes tamaños del modelo T5 (small, base y large) para analizar el compromiso entre el rendimiento predictivo y el costo computacional, buscando el equilibrio óptimo para una implementación práctica.

\section{Diseño del bot}

El bot de trading se diseñó con una arquitectura modular para facilitar su configuración, mantenimiento y extensibilidad. Se divide en cuatro componentes principales que operan de forma desacoplada:

\begin{enumerate}
    \item Orquestador: es el punto de entrada de la aplicación. Su función es instanciar, configurar y conectar los demás componentes, iniciando el ciclo de operación del bot.
    \item Motor del Bot: contiene el bucle principal de ejecución. En cada iteración, orquesta el flujo de trabajo: solicita los datos más recientes del mercado al módulo de intercambio, pasa estos datos al módulo de estrategia para obtener una señal de trading y, si la señal lo indica, instruye al módulo de intercambio para que ejecute la orden correspondiente.
    \item Módulo de Exchange: abstrae toda la comunicación con la API del exchange de criptomonedas. Es responsable de obtener datos de mercado (precios, volúmenes) y de ejecutar órdenes (compra, venta). Su diseño intercambiable permite adaptar el bot a diferentes plataformas (por ejemplo, Binance, KuCoin) con mínimas modificaciones.
    \item Módulo de Estrategia: carga el modelo de machine learning pre-entrenado (en este caso, el ensamble de Chronos y XGBoost) y lo utiliza para procesar los datos de mercado y generar una señal de trading: comprar, vender o mantener.
\end{enumerate}

En esencia, el bot opera en un ciclo continuo: recolecta datos, predice una señal con el modelo y actúa sobre esa predicción. Esta separación de responsabilidades garantiza que la lógica de la estrategia de trading esté completamente aislada de los detalles de implementación de la comunicación con el mercado.
